## 第 1 章 · 使用说明

### 1.1 难度分级
- **L1**（基础）：任何 AI 工程师都该会的，答错就直接挂
- **L2**（决策追问）：考验你为什么这么选、还考虑过什么
- **L3**（拷打底裤）：考验思维深度与边界感，答不上不丢人但答得上是亮点

### 1.2 每题统一结构
- **意图** —— 面试官想验证什么
- **答题骨架** —— 主答案，30 秒能讲完的核心
- **加分项** —— 让你从合格到出彩
- **避雷点** —— 别说的话/容易翻车的回答
- **反问/引导** —— 把话题拉回自己擅长的方向

### 1.3 通用心法
- 凡是说"我这么做"，必须配套"还考虑过 X，没选是因为 Y"
- 凡是说改进效果，必须给数字（哪怕粗略）
- 凡是说"未解决"，必须有下一步计划
- 不会的不装会，但会把不会的方向引到自己会的相邻领域

## 第 2 章 · 技术选型快答（不展开）

> 目的：被问到能 1–2 句答完不卡壳。深挖再展开决策依据。

### 2.1 LanceDB vs Qdrant / Milvus / pgvector

> **LanceDB**：嵌入式（无独立服务进程）+ Arrow 列式存储 + Python 原生易集成。团队规模小、无专职运维，当前数据量万级 chunk，LanceDB 足够；规模上去后抽象层已留好，可平滑迁移到 Qdrant。Milvus 重，pgvector 在 ANN 上不如专用引擎。

### 2.2 Meilisearch vs Elasticsearch / OpenSearch

> **Meilisearch**：开箱即用、中文分词够用、运维零成本。ES/OS 功能强但运维重、内存占用大，对当前规模 over-engineering。如果未来需要复杂查询 DSL、多字段聚合，再考虑 OpenSearch。

### 2.3 为什么不用 LangChain / LlamaIndex

> **三个原因**：① 抽象层叠太多，调试时栈深 5–6 层难定位；② 默认实现（如 RecursiveCharacterTextSplitter）不适配规范文档的章节层级；③ 我们流水线里很多"边走边发现的"小修补（入库前过滤、动态阈值），框架反而约束。理解其高级 RAG 模式（Parent-Document、Multi-Query、HyDE）并选择性借鉴。

### 2.4 为什么 RAG 用 Python 而主系统是 Node.js

> Python 在 ML/embedding/PDF 解析的生态明显成熟（pdfplumber、PyPDF2、numpy/scipy 后处理）。RAG 与主系统通过 HTTP+SSE 解耦，独立部署、独立扩缩容、独立迭代，团队边界清晰。Node 强在 IO 与前端协作，让它做对话编排正合适。

### 2.5 Qwen embedding vs Doubao

> 详见第 5.1 故事。一句话：Doubao 在建筑规范这种**单领域语料**下出现严重各向异性，cosine 分数虚高失去区分度。Qwen text-embedding-v4 向量空间天然各向同性，**无需后处理**直接可用，TOP5 命中率从 4% 提升到 96%。

## 第 3 章 · 工程实现快答（不展开）

### 3.1 FastAPI async / SSE 流式

> 全链路 async（httpx 替代 requests，aiomysql/asyncpg 替代同步驱动）。SSE 走 `StreamingResponse` + async generator，Nginx 关闭 `proxy_buffering`。LLM 调用直接 `async for chunk in stream` 转发到客户端。

### 3.2 Redis 限流策略

> 滑动窗口（Sorted Set + 时间戳），按 `user_id` 维度。无 Redis 时降级为内存 LRU + 单进程限流，保证开发环境可用。

### 3.3 双索引一致性

> 写入路径：先写 MySQL（事务）→ 再写 LanceDB → 再写 Meilisearch。任一步失败，向后回滚（删已写入项 + 标记 chunk 为 `index_failed`）。后台有补偿任务定期扫描 `index_failed` 重试。**最终一致而非强一致**——业务可接受短时间检索缺失，不可接受丢数据。

### 3.4 后台文档处理（BackgroundTasks 局限）

> 当前用 FastAPI BackgroundTasks，进程重启会丢任务。**已知短板**，下一步替换为 ARQ（轻量 Redis 队列）或 Dramatiq。没上 Celery 是因为对当前规模过重、运维成本高。

### 3.5 错误重试与退避

> embedding/LLM 调用：指数退避 + 抖动，最多 3 次。区分错误类型——4xx 不重试（请求本身错的，重试无意义）、5xx 与超时重试、429 触发退避并降级（如切到备用 provider）。

## 第 4 章 · RAG 系统设计深挖

### 4.1 完整讲一遍你的 RAG 流水线（L1）

**意图** —— 验证你能不能 30 秒讲清整个系统的形状（结构感）。

**答题骨架**：
> 四段式。① **数据处理**：PDF → Markdown → LLM 辅助清洗 → 树结构切片 → 入库前过滤 → 双索引（向量 + 全文）；② **检索**：向量 + BM25 并行召回 → RRF 融合 → 三级去重 → 动态阈值 → 召回增强（兄弟/子/祖先节点）→ Token 预算限流；③ **上下文组装**：按树路径拼接 + 标题路径保留（便于引用追溯）；④ **评测**：金标集 + Score Spread 指标，规范类自动评、案例类人评。

**加分项**：
- 主动把"评测"作为独立一段，而不是隐藏在调参里
- 提到"案例推荐 vs 规范查询"两类 KB 的差异化处理（指向第 6 章）
- 一句话点出技术栈：FastAPI + LanceDB + Meilisearch + Qwen v4 embedding + Qwen-plus 生成

**避雷点**：
- 别一上来就报技术栈，结构感先于工具
- 别画 LangChain 标准流程图（容易被怀疑照搬）

**反问/引导**：
> "如果你想，可以挑任意一段我们细看，我都能讲到决策依据。"——把节奏交回自己。

### 4.2 为什么用"树结构切片"而不是固定 size 切？（L2）

**意图** —— 验证你是因为业务特点选的方案，还是无脑套默认。

**答题骨架**：
> 因为建筑规范天然是层级结构（章/节/条），固定 size 切会把 "3.2.1 防火等级" 切到一半，破坏完整语义单元。我按 Markdown 标题层级建 TreeNode 树，材料化路径 `0001/0002/0003` 保留父子关系。这样做有两个收益：① 切片边界天然对齐语义；② 后续可做兄弟/子节点扩展（4.5 题）和上下文组装时拼回章节路径。

**加分项**：
- 主动指出**反例**：纯案例文档（无层级）退化为按段落切，没强行套树
- 解释材料化路径而不用邻接表/嵌套集——查询单条 SQL 即可拿到所有祖先/后代

**避雷点**：
- 别说"树结构更先进"——没有更先进，只有更适配
- 别忽略超长章节的处理（超长章节再按 token 二次切，阈值 800 token）

**反问/引导**：
> "这套切片对规范类很匹配，但对案例类要不要也强行树化，我们内部讨论过，最终选了不强行——可以展开聊。"

### 4.3 "入库前过滤"这一层的设计依据是什么？阈值怎么定？误杀怎么办？（L2）

**意图** —— 验证你是不是真在迭代中发现的、还是抄来的固定规则。

**答题骨架**：
> **怎么发现的**：测试时反复看到 top_k 里出现 "3.2.1"、"概述" 这种纯标题 chunk，挤占名额。**诊断**：切片器对短章节产出仅含标题的 chunk；表格被切碎产生大量碎片；部分文档存在长串英文标识符块。**规则**：① 字符长度 < 50；② 仅包含标题/编号/纯英文 token；③ 信息密度过低（重复字符比例 > 40%）。**阈值怎么定**：抽样 200 个被过滤 chunk 人工 review，调到误杀率 < 5%。**误杀兜底**：保留过滤日志，在 KB 层面提供"召回失败"的反查工具，发现是过滤误杀就把规则放宽。

**加分项**：
- 强调"人工抽样验证"是过滤规则上线的前置条件
- 提到过滤后 top_k 命中率提升的数字（虽然没单独 ablation 但能作为综合改进的一部分）

**避雷点**：
- 别说"按经验定的阈值"，要给出验证方式
- 别说"过滤永远不误杀"——任何过滤都有 trade-off

**反问/引导**：
> "这个故事其实是我从'凭感觉调参'转向'指标驱动'的起点之一，可以展开聊整个心态变化。"

### 4.4 Hybrid 检索为什么必要？纯向量不够吗？（L2）

**意图** —— 验证你能不能拿出**反例**，而不只是说"两者结合更好"。

**答题骨架**：
> 反例：用户问 "GB 50108-2008 第 3 条怎么规定" —— 这种带规范编号的查询，纯向量召回非常差，因为通用 embedding 对编号串区分度极低（"GB 50108" 和 "GB 50118" 在向量空间近乎重合）；但 BM25 直接命中编号 token，TOP1 准确。反过来纯 BM25 处理 "防火设计的基本要求" 这种语义查询又召不回包含 "建筑防火设计原则" 的条款。所以必须 Hybrid：向量擅长语义、全文擅长术语和编号。融合用 RRF 而非加权和——因为向量 cosine 和 BM25 分数量纲完全不同，RRF 不依赖分数尺度。

**加分项**：
- 主动给一个真实反例（编号类）
- 提到 RRF 的 k 值（默认 60）是怎么定的，以及为什么不调

**避雷点**：
- 别只说"互补"——要给具体反例
- 别把 RRF 公式背错（`score = Σ 1/(k + rank_i)`）

**反问/引导**：
> "如果你想深挖，我也能讲为什么我们没在 Hybrid 后加 Cross-Encoder reranker——是 ROI 和延迟的取舍。"

### 4.5 召回增强（兄弟/子/祖先）什么时候反而有害？（L3）

**意图** —— 验证你有没有"任何收益策略都有边界"的工程意识。

**答题骨架**：
> 三种典型反例：① **跨章节误扩展**：命中 "3.2.1" 时取兄弟 "3.2.2"，但兄弟讨论的是完全不相关的子主题，反而稀释主题；② **token 预算被吃光**：长章节有 20 个兄弟，全取会把 LLM 上下文撑爆；③ **主题漂移**：取祖先节点 "第三章 防火" 时引入与查询无关的章首总览，让 LLM 抓不住重点。**对应防护**：① 兄弟数量上限（默认 2 个）；② 总 token 预算硬限制（4.7 题）；③ 祖先只取直接父节点的标题路径而非全文。

**加分项**：
- 主动列出三种边界，体现"系统化思考"而非"想到一个算一个"
- 提到这些规则是从 bad case 反推出来的，不是先验设计

**避雷点**：
- 别说"召回增强永远有用"
- 别忽略"取多少"这个超参怎么调的

**反问/引导**：
> "这其实是个典型的'好策略走过头'的案例，可以聊我们如何在评测中发现它的副作用。"

### 4.6 动态阈值的具体算法是什么？为什么不用固定值？（L3）

**意图** —— 验证你能不能讲清算法逻辑，而不只是说"用了动态阈值"。

**答题骨架**：
> **为什么不固定**：不同 query 召回的分数分布天差地别——精确查询 TOP1 可能 0.85、TOP10 还有 0.7；模糊查询 TOP1 才 0.5。固定阈值在前者太宽松（噪声进来）、在后者太严苛（漏召回）。**算法**：取召回结果分数序列，计算 `mean - α * std`（α 默认 1.0），低于此值的截断；同时保底——至少保留 top 3 防止全被截。**调优过程**：α 在 [0.5, 2.0] 上做了网格搜索，跑 5 道金标题观察 Score Spread，α=1.0 综合最优。

**加分项**：
- 给出"保底 top 3"这种工程兜底，体现实战经验
- 主动说局限：单峰分布假设，多峰时退化
- 提到这个算法在不同 KB（规范 vs 案例）上的 α 不同

**避雷点**：
- 别把算法说得过于"高大上"——本质就是 z-score 截断
- 别忽略"保底"机制，否则面试官会立刻问"如果整体分数都低呢"

**反问/引导**：
> "更复杂的阈值算法（如 OTSU 大津法）我评估过，对当前规模 ROI 不高，可以聊取舍逻辑。"

### 4.7 为什么用 Token 预算而不是 chunk 数控制上下文长度？（L2）

**意图** —— 验证你理解 LLM 上下文的本质单位。

**答题骨架**：
> 因为 LLM 的硬限制是 token 而不是 chunk 数。我们的 chunk 长度差异极大（100 字到 800 字都有），用 chunk 数控制会出现两种极端：要么 8 个短 chunk 才用了 1k token 浪费窗口，要么 8 个长 chunk 直接撑爆 16k 限制。**实现**：召回排序后按预算从高分往下加，超预算的丢弃；同一树节点的相关上下文（标题路径）单独计入预算，保证可追溯性优先。

**加分项**：
- 提到"树路径保留"在 token 预算里有独立配额（不挤占召回内容）
- 提到不同 LLM 的 token 计数差异（用 tiktoken 估算 + 留 10% 缓冲）

**避雷点**：
- 别说"chunk 数也行"——给出反例才有说服力
- 别忘记输出 token 也要扣预算

**反问/引导**：
> "Token 预算分配本身也有学问——比如召回 vs 历史对话的预算分配，我们也讨论过。"

### 4.8 上下文组装为什么要带"标题路径"？（L2）

**意图** —— 验证你理解"可追溯性"在 RAG 中的工程价值。

**答题骨架**：
> 三个目的：① **降低幻觉**：LLM 看到 "《建筑设计防火规范 GB 50016》5.3.1 防火分区" 这样的明确出处，更倾向于在生成时引用而非编造；② **回答可引用**：建筑师拿到答案后能直接告诉客户"这是 GB 50016 第 5.3.1 条"，避免"AI 大概说"；③ **bad case 复盘**：用户反馈错时能直接定位是哪条规范召回错了。**实现**：拼接格式 `[《xxx规范》→ 第 N 章 → 5.3.1 防火分区]\n{正文}`。

**加分项**：
- 强调"可追溯"是建筑领域的硬需求（设计要扛监管和审图）
- 提到 prompt 中显式要求 LLM "引用 chunk 时带上方括号里的路径"

**避雷点**：
- 别说"为了好看"——要落到业务价值
- 别忽略 prompt 侧的配合（光给路径不要求引用，LLM 会忽略）

**反问/引导**：
> "可追溯也是我们和建筑师协作时的强约束，可以聊跨专业沟通的故事。"

### 4.9 你这套设计在 100 万文档规模下哪里会先崩？（L3）

**意图** —— 验证你对系统瓶颈有真实认识，而不是"我的系统什么规模都行"。

**答题骨架**：
> 三个先崩点。① **LanceDB 单机 ANN 索引**：当前嵌入式部署，向量数 > 几百万后查询延迟陡增，需要切到 Qdrant cluster 或 Milvus；② **Meilisearch 内存压力**：全文索引常驻内存，百万文档量级单机吃不消，要么分片要么切 OpenSearch；③ **Embedding 限速**：Qwen API 有 QPS 限制，批量 ingest 100 万文档要排队几天，需要本地化部署 bge-m3 或类似模型。**额外的隐性瓶颈**：MySQL 上 chunk 表（千万行级）的全表扫描——当前 `delete_time IS NULL` 这种过滤会变慢，需要优化索引和分区。

**加分项**：
- 主动区分"显性瓶颈"（向量/全文索引）和"隐性瓶颈"（MySQL chunk 表）
- 给出**迁移路径**而不只是"不行了"
- 提到 ingest 和 serve 应该分离部署，避免互相影响

**避雷点**：
- 别说"加机器就行"——具体说哪部分加什么
- 别避开"我没经验过百万规模"，可以坦诚加一句"以下是基于推算"

**反问/引导**：
> "如果你那边真有这个量级的需求，我很想听听你们怎么解决的。"——把姿态放成请教。

### 4.10 如果让你从零重做这个 RAG 系统，会怎么改？（L3）

**意图** —— 验证你对项目的反思深度，以及是否真的在持续学习。

**答题骨架**：
> 三个最重要的改动。① **评测先行**：金标集和评测脚本作为第一个里程碑，而不是"等系统跑通了再说"——这是我们当前最大短板，所有调参没有 CI 保护；② **Reranker 一开始就上**：bge-reranker-v2-m3 在 Hybrid 后做精排，预期能进一步把 TOP1 准确率从 100%（5 题）拉到更稳健的水平（更大评测集上）；③ **ingest 与 serve 物理分离**：当前同进程内做文档处理和检索，文档高峰期会影响检索延迟，下一版要拆成两个服务。**保留**：树结构切片 + Hybrid + 入库前过滤这三件事是稳赚的，不动。

**加分项**：
- 区分"会改"和"不会改"——后者体现你对自己工作的判断
- 把"评测先行"放第一位，体现工程成熟度
- 不卷"什么新技术都要用"，理性谈 ROI

**避雷点**：
- 别说"全推倒重来"——会显得对原方案不自信
- 别说"加 Agent"——除非你能讲清为什么

**反问/引导**：
> "重做的优先级排序本身就是个判断题，我可以聊我为什么把评测排第一。"

## 第 5 章 · 搭建过程踩坑故事池

> **统一结构**：怎么发现 → 诊断 → 备选方案 → 落地 → 数字效果 → 现在看的不足。

### 5.1 Embedding 各向异性，cosine 失效（L3 · 招牌故事）

**怎么发现的**：
> 测试规范类查询时发现一个反常现象——所有 query 召回结果的 cosine 分数都虚高在 0.87–0.92 之间，正确 chunk 和无关 chunk 的分数差几乎为零。我把 threshold 从 0.85 调到 0.90，正确答案直接被滤掉；调到 0.80，噪声爆炸。**典型的"好像哪里都不对"的状态**。

**诊断思路**：
> 第一步排除单个 query 的偶然——批量跑 20 个 query，分布一致；第二步排除 chunk 质量——人工抽 5 对（正确 vs 噪声）肉眼能区分，说明语义是有差异的，是 embedding 表达不出来；第三步开始查文献，定位到 **anisotropy（各向异性）** 这个学术概念：在窄领域语料下，embedding 模型产出的向量在空间中聚集到一个主导方向，cosine 相似度普遍虚高。

**备选方案**：
> ① **All-but-the-Top（AbTT）后处理**：去除前 N 个主成分。试了——绝对分数从 0.87 降到 0.30，看似合理，但 Score Spread（正确 chunk 分数 − TOP1 噪声分数）≈ 0，**排序仍然无意义**，降阈值后精确率崩。结论：AbTT 是缓解，不是解决。
> ② **领域微调**：理论上有效，但需要标注数据 + 训练资源 + 评测框架，应届团队短期 ROI 太低。
> ③ **换模型**：成本最低、效果立竿见影。最终选这个。

**落地**：
> 切换到 Qwen text-embedding-v4（1024 维），向量空间天然各向同性，无需任何后处理直接可用。

**数字效果**：
> 5 道金标题（top_k=5、threshold=0.0、RRF 0.7/0.3）：
> - 向量 TOP5 命中：1/25 (4%) → 24/25 **(96%)**
> - Hybrid TOP1 准确率：3/5 (60%) → 5/5 **(100%)**
> - Score Spread：≈ 0 → > +0.5
> - LLM 端到端准确率：N/A → 5/5

**现在看的不足**：
> ① 5 道金标题样本太小，结论方向性可信但泛化性需要更大样本验证；② 没做不同 KB 的对比测试——案例 KB 上 Qwen v4 是否同样优秀，没系统验证过；③ 当时没意识到这是个普遍现象，后来发现 BERT 系列、SimCSE 论文里都有讨论各向异性，应该更早查文献。

**面试用法**：
- 这是招牌故事，"最有挑战的问题"必抽
- 体现"发现问题 → 根因诊断 → 对照实验 → 决策落地"完整闭环
- 主动暴露不足比假装完美更体现成熟度

### 5.2 短标题/纯标号挤占 top_k → 入库前过滤层（L2）

**怎么发现的**：测试时反复看到检索结果里出现 "3.2.1"、"概述" 这种纯标题或十几字的极短 chunk，挤占 top_k 名额。一开始以为是个例，后来统计发现规范类 KB 里此类 chunk 占比 8–12%，相当严重。

**诊断**：① 切片器对短章节产出仅含标题的 chunk；② 表格被切碎产生大量 < 30 字的碎片；③ 部分文档存在长串英文标识符块（如规范引用列表）。

**备选方案**：
> ① 改切片器逻辑——风险大，规则一改全量重建索引；
> ② 检索后过滤——浪费 top_k 名额，治标不治本；
> ③ **入库前过滤层**——独立模块、规则可调、上线风险小。选 ③。

**落地**：在 embedding 之前加过滤层：长度 < 50 字 / 仅含标题或编号 / 信息密度过低（重复字符比例 > 40%）三条规则，任一命中即丢弃。

**数字效果**：过滤后 top_k 中无效 chunk 占比从 8–12% 降到 < 1%；与其他改动叠加后整体召回质量明显改善（无单独 ablation）。

**现在看的不足**：① 没做单独 ablation，无法量化"过滤本身贡献多少召回提升"；② 阈值是抽样人工调的，没有自动化校准；③ 信息密度规则对中英混排文本判定不准，存在少量误杀。

### 5.3 PDF→MD 章节编号丢失 → LLM 辅助清洗（L2）

**怎么发现的**：跑测试时发现召回的 chunk 里编号常常是错的——比如规范实际是 "5.3.1"，转出来变成 "5.31" 或被识别成普通段落 "5 . 3 . 1" 三个独立 token。LLM 引用时直接把错误编号也带出去，建筑师当场指出 "这条编号根本不存在"。

**诊断**：PDF 排版中的编号有空格、换行、字体差异，pdfplumber 默认输出会破坏结构。直接修工具链改动太大、收益不确定。

**备选方案**：
> ① 写正则规则修编号——脆弱，规范格式多样；
> ② 换 PDF 解析工具（Marker / unstructured）——重写工程代价大，不一定彻底解决；
> ③ **LLM 辅助清洗**：把转出的 MD 喂给 LLM，要求"恢复章节层级与编号"，作为二次清洗步骤。选 ③。

**落地**：清洗 prompt 包含——① 输出严格 Markdown 格式；② 编号必须保留原文形式（"5.3.1" 不能写成 "5.3.1节"）；③ 同时为每个章节生成简短摘要作为 chunk 元数据。

**数字效果**：编号正确率从约 70% 提升到 > 95%；摘要作为辅助索引让某些"语义说法不同但语义一致"的查询召回率提升。

**现在看的不足**：① LLM 清洗成本高（每篇文档要全文过一遍 LLM），ingest 速度受限；② 偶有 LLM 改写时多删/少删一行内容的幻觉，需要后处理校验；③ 当前 prompt 是手调的，没系统化评估过不同 prompt 的清洗质量。

### 5.4 召回断章取义 → 召回增强（L2）

**怎么发现的**：建筑师反馈"答案不完整，规范条款总像被掐头去尾"。复盘时发现 LLM 拿到的 chunk 经常是孤立条款，缺失上下文，比如命中 "3.2.1 防火等级 一级" 但不知道这是哪一节、对应哪种建筑类型。

**诊断**：树切片做对了（按章节切），但检索出来的是孤立叶节点。LLM 没有上下文判断条款的适用范围。

**备选方案**：
> ① 切片时强制带父节点信息——会让所有 chunk 都很冗长，浪费向量空间；
> ② 检索后回取——按需取祖先和兄弟，灵活；
> ③ 直接加大 chunk size——治标不治本，且重新触发各种 4.5 题里的副作用。选 ②。

**落地**：命中 chunk 后自动回取——① 直接父节点的标题路径（不取正文）；② 同父下的兄弟节点（最多 2 个）；③ 子节点全文（如果命中的是中间层节点）。

**数字效果**：建筑师对"答案完整度"的主观打分明显提升；具体数字没量化（这正是案例推荐评测难做的体现，见第 6 章）。

**现在看的不足**：① 兄弟取数固定为 2 是经验值，没按 KB 调；② 跨章节扩展边界没卡死，偶尔会引入主题不相关内容（4.5 题的副作用）；③ 没做"取了之后是否真的帮助 LLM"的反向验证。

---

### 5.5 父子节点同时进上下文 → 三级去重（L2）

**怎么发现的**：上 4.5 召回增强后出现新问题——Hybrid 召回同时命中父节点（章节摘要）和子节点（具体条款），LLM 上下文里出现重复内容，token 翻倍且 LLM 困惑（"为什么有两份相似的内容"）。

**诊断**：召回增强 + 双索引并行召回的组合效应——向量召回了父节点、BM25 又召回了子节点，融合后两个都进了去重前结果。

**备选方案**：
> ① 检索阶段就跳过父子关系——会漏召回；
> ② 上下文阶段去重——有损但安全。选 ②。

**落地**：三级去重——① **ID 去重**：同一 chunk 不重复；② **内容相似度去重**：embedding 相似度 > 0.95 的去重；③ **父子关系去重**：保留子节点（信息更具体），丢弃同链路祖先节点的正文（保留路径）。

**数字效果**：上下文 token 平均下降 20–30%；LLM 回答中"重复表述"问题消失。

**现在看的不足**：① 内容相似度去重的阈值（0.95）是经验值，没系统校准；② 父子去重在某些场景反而有害——比如父节点有重要总览信息，目前粗暴丢弃。

---

### 5.6 固定阈值跨 query 表现差 → 动态阈值（L3）

**怎么发现的**：调参时发现一个矛盾——同一 threshold（0.5）在 query A 上效果好，在 query B 上要么过严（漏召回）要么过宽（噪声），始终找不到一个全局最优值。

**诊断**：不同 query 的召回分数分布天差地别——精确查询 TOP1 0.85、TOP10 0.7；模糊查询 TOP1 才 0.5。固定阈值无法兼顾。

**备选方案**：
> ① 按 query 类型分类用不同阈值——分类本身就难且不稳定；
> ② **按当前召回结果的分数分布自适应**——简单且自洽。选 ②。

**落地**：见 4.6 题——`mean - α * std` + 保底 top 3。

**数字效果**：不同 query 之间的召回质量方差明显下降；整体 Score Spread 提升。

**现在看的不足**：① 单峰分布假设，多峰场景退化；② α 没按 KB 调；③ 保底 top 3 在极端低质量召回场景会"硬塞噪声"。

---

### 5.7 top_k 截断丢失重要内容 → Token 预算分配（L2）

**怎么发现的**：top_k=5 时偶尔出现"第 6 条恰好是关键答案"的反例，调高 top_k 又灌进噪声，调不准。

**诊断**：本质是 chunk 长度差异导致 top_k 这个度量失效——5 个短 chunk 和 5 个长 chunk 占用 token 差 5 倍。

**备选方案**：① 增大 top_k 同时加 reranker 截断——多一层组件；② **直接按 token 预算累加召回结果**——简单直接。选 ②。

**落地**：见 4.7 题。

**数字效果**：上下文利用率（实际 token / 预算 token）从约 40% 提升到 70–80%；漏掉关键答案的反例消失。

**现在看的不足**：① 预算分配在召回 vs 历史对话之间的拆分还是经验值；② 没做"如果给 LLM 更多预算，质量是否单调上升"的实验。

---

### 5.8 规范编号语义检索失败 → Hybrid 起源（L2）

**怎么发现的**：早期纯向量方案上线，建筑师立刻反馈："我搜 'GB 50108-2008'，搜到的全是其他不相关的规范条款，这肯定不对。"

**诊断**：通用 embedding 对编号串的区分度极低——"GB 50108"、"GB 50118"、"GB 50068" 在向量空间近乎重合。这是 embedding 模型的固有局限，不是参数能调出来的。

**备选方案**：
> ① 编号字段单独建索引，检索时拼接结果——工程上 hack；
> ② **引入全文检索（BM25）**，term 匹配天然解决编号问题——优雅。选 ②。

**落地**：上 Meilisearch + RRF 融合，形成双索引 Hybrid 检索流水线。

**数字效果**：编号类查询从近乎不可用变成 TOP1 命中；正文类查询保持稳定。

**现在看的不足**：① 全文索引和向量索引的同步延迟没监控；② RRF 权重 0.7/0.3 是经验值，没按 KB 类型分别调。

### 5.9 表格/图片中数值召不回（未解决）（L2 · 坦诚不足）

**怎么发现的**：测试题 Q3 "锅炉房距住宅最小防火间距是多少米？" — 召回完全失败。复盘发现答案在规范的**表格图片**中，PDF→MD 转换时表格被丢弃或转成乱码。

**诊断**：① pdfplumber 对表格识别能力弱；② 即使能识别，表格被切成"单元格 chunk"后失去行列结构，BM25 和向量都拿不到完整语义。

**当前规划**：
> ① 引入 PDF 布局解析（PP-StructureV2 / Marker / unstructured）；
> ② 表格 OCR 后结构化为 Markdown 重新入库；
> ③ 对图表生成自然语言描述作为辅助索引。
> **没立刻做**是因为引入新组件会重写整条 ingest 链路，需要先稳定评测体系（5.10）才能验证改动效果。

**面试用法**：主动暴露的"未解决项"，体现工程节奏感（不为了完美而堆栈）。

---

### 5.10 评测靠人工，无法快速回归（未解决 · 当前最大短板）（L3）

**怎么发现的**：每次调参后只能跑一次手工对比，没法快速验证多组参数。某次 RRF 权重调整后忘了跑全量评测，第二天才被建筑师发现某类 query 退化。

**诊断**：评测脚本不完整、金标集太小（5 道）、无 CI 触发机制。

**当前规划**：
> ① 扩充金标集到 30+ 题，覆盖三类业务场景（规范/案例/其他专业）；
> ② 引入 RAGAS 指标：faithfulness、context_precision、context_recall；
> ③ 评测脚本化、PR 触发、形成"调参 → 评测 → 决策"闭环；
> ④ 沉淀 Bad Case 库，定期复盘。

**为什么是"最大短板"**：
> 没有评测体系保护，所有调参都是赌博——调好了不知道为什么好，调坏了不知道为什么坏。这是当前一切下游改进（reranker、HyDE、领域微调）的前置依赖。

**面试用法**：被问"现在最想做什么改进"时优先答这个，体现工程优先级判断力。

## 第 6 章 · 与建筑师的协作（聚焦"案例推荐"主观评估）

> **核心矛盾**：两类知识库的评测难度天差地别——
> - **规范查询**：有标准答案，可以做自动化评测，建筑师只参与出题；
> - **案例推荐**：相似度天然主观（设计风格、空间逻辑、客户类型、地域气候等多维度），**没有标准答案**，必须靠建筑师评判。
>
> 这一章所有故事的发生场景都是案例推荐 KB。

---

### 6.1 你们团队是怎么协作的？建筑师扮演什么角色？（L1）

**意图** —— 验证你的协作经验是真实的，还是 PPT 上写写。

**答题骨架**：
> 团队里有 4–5 位建筑专业成员。我们的 RAG 有两类知识库——**规范查询**和**案例推荐**。这两类的协作模式完全不同：
> - 规范查询：建筑师参与出题（提供"用户会怎么问"），但评测能自动化（标准答案是规范条款本身）；
> - 案例推荐：建筑师**就是评测者**，因为"这个案例和用户问题匹配吗"是个主观判断，没有标准答案。
>
> 我和建筑师的协作 90% 时间都花在案例推荐 KB 的精度提升上。

**加分项**：
- 主动指出"两类 KB 的评测难度差异"——体现你抽象出了协作模式
- 提到具体人数（4–5 位）让回答有体感

**避雷点**：
- 别说"和建筑师一起优化系统"——太空泛
- 别把建筑师角色说成"提需求的人"——他们是评测者，是合作方

**反问/引导**：
> "案例推荐评测难做，是这个项目里我学到最多的事——可以聊我们怎么设计这套人评流程的。"

---

### 6.2 为什么案例推荐没法做自动化评测？（L2）

**意图** —— 验证你能不能讲清"主观性"在工程上的具体表现。

**答题骨架**：
> 因为"两个建筑案例是否相似"涉及多个维度：建筑功能（住宅/商业/医院）、设计风格（现代/古典/解构）、空间逻辑（流线/采光/动静分区）、地域气候（南方/北方/高原）、客户类型（政府/开发商/私人）、设计阶段（方案/扩初/施工图）。**这些维度的权重因人而异，更因 query 的隐含意图而异**。
>
> 举例：用户问 "想看一个采光做得好的住宅设计"——
> - 候选 A：北方住宅，大窗 + 中庭，采光优秀；
> - 候选 B：南方住宅，遮阳 + 通透，光线柔和；
>
> 都"采光好"，但好的方向完全不同。建筑师能根据用户的隐含上下文判断哪个更合适，**自动化指标做不到**。

**加分项**：
- 给具体例子（A vs B）——让"主观性"具象化
- 提到"隐含意图"是关键——RAG 系统看不到用户的真实场景

**避雷点**：
- 别说"加 LLM 评判就行"——LLM 评判在这种深度专业领域并不可靠
- 别避谈"主观性"——这是事实，不丢人

**反问/引导**：
> "这个问题其实让我对'什么是好检索'的理解发生了变化——可以聊指标设计上的反思。"

---

### 6.3 在没有标准答案的情况下，你怎么衡量案例推荐的"精度"？（L2）

**意图** —— 验证你有没有在"无客观指标"的环境下设计评测体系的能力。

**答题骨架**：
> 三层做法：
> ① **多人交叉打分**：同一组（query + 召回结果）让 2–3 位建筑师独立 1–5 分打分，取平均；
> ② **一致性度量**：计算建筑师之间的打分一致性（用 Krippendorff's α 简化版），如果一致性低（α < 0.5）说明这道题本身定义不清，重新设计；
> ③ **多维度评分而非单分**：把"召回质量"拆成 3 个子维度——主题相关性、设计水平相关性、客户场景相关性——分别打分，避免一个总分什么都说不清。
>
> **结果不是绝对的"准确率"，而是版本之间的相对提升**。比如改前平均 3.2 分、改后 3.8 分，方向就够了。

**加分项**：
- 提到"一致性度量"——体现你考虑过评测者噪声
- 提到"多维度评分"——体现你会拆解模糊指标
- 提到"相对提升而非绝对值"——体现你理解评测的实际用途

**避雷点**：
- 别说"建筑师打分就够了"——单人打分有偏差
- 别说"我不会评测主观系统"——这是机会展示你设计了什么

**反问/引导**：
> "这套人评流程其实是迭代出来的——一开始用过更简单的方法，被建筑师反馈推翻过，可以聊那个故事（指向 6.8）。"

### 6.4 建筑师反馈"这个案例不对"时，你怎么追问到底是召回错了还是 LLM 总结错了？（L3）

**意图** —— 验证你能不能跨专业沟通，把"主观感受"翻译成可定位的技术问题。

**答题骨架**：
> 我会把回答**拆成两层**让建筑师分别评：
> ① "我们召回了这 5 个案例，你觉得这 5 个里有几个**值得给用户看**？" — 这一层评的是**召回质量**；
> ② "在你认为值得看的这几个里，LLM 的总结描述准确吗？有没有夸大或漏掉关键点？" — 这一层评的是**生成质量**。
>
> 这样能把"答案不对"细分到三种可能：
> - 5 个都不行 → **召回失败**，回去查检索流水线；
> - 召回里有合适的、但 LLM 没突出 → **生成失败**，回去调 prompt；
> - 召回 OK、生成 OK，但建筑师还是觉得不对 → **意图理解失败**，可能是上层路由把请求送错了 KB。
>
> 这个分层追问的习惯是**建筑师教会我的**——他们一开始反馈都是"答案不对"，是反复沟通后才能稳定地按这个结构反馈。

**加分项**：
- 把"分层定位"作为方法论讲，而不是个例
- 主动承认"是建筑师教会我的"——体现谦逊和学习能力
- 第三种"意图失败"展示你理解 RAG 上下游边界

**避雷点**：
- 别说"我自己看 log 排查"——核心是**和建筑师对话的方式**
- 别把锅甩给上游路由——要展示你能主动协助定位

**反问/引导**：
> "这种'技术翻译'的能力其实是项目里最难学的，可以聊我犯过的几个沟通错误。"

### 6.5 案例推荐和规范查询用的是同一套检索流水线吗？为什么？（L3）

**意图** —— 验证你对"业务差异如何映射到技术参数"的理解。

**答题骨架**：
> **同一套流水线，不同参数与策略**。底层架构（数据处理 → Hybrid 检索 → 上下文组装）完全共享，差异在配置层：
>
> | 维度 | 规范查询 | 案例推荐 |
> |---|---|---|
> | RRF 权重 | 0.5 / 0.5（语义和编号都重要）| 0.7 / 0.3（语义为主，全文兜底）|
> | top_k | 5（精准为主）| 10（多样性优先，建筑师在前端再筛）|
> | 召回增强 | 强（兄弟+祖先全取）| 弱（案例之间无层级关系）|
> | 入库前过滤 | 严格（短块全杀）| 宽松（短描述也可能是有价值的标签）|
> | 树切片 | 强制（章节层级清晰）| 退化为段落切片 |
> | 评测方式 | 自动化（标准答案）| 人评（建筑师打分）|
>
> **为什么不拆成两套系统**：① 维护成本翻倍；② 共用流水线让改进可以同时受益（比如 5.5 三级去重对两类 KB 都有效）；③ 业务上未来可能加更多 KB 类型，统一流水线 + 配置化更可持续。

**加分项**：
- 给出表格对比，体现"差异化"是有具体颗粒度的
- 解释"不拆系统"的原因——体现工程判断
- 主动指出未来扩展性

**避雷点**：
- 别说"完全一样"或"完全不同"——业务差异化必然有，但不是推倒重来
- 别忽略评测方式的差异——这才是 6.5 题的灵魂

**反问/引导**：
> "这种'统一流水线 + 配置化差异'的设计模式我觉得是 RAG 系统设计里的关键直觉之一，可以聊我是怎么意识到的。"

### 7.1 协议层：HTTP + SSE 接入时要注意什么？（L2）

**意图** —— 验证你做过真实的 SSE 接入，知道里面的坑。

**答题骨架**：
> 三个最容易踩的坑：
> ① **Nginx / Node 中间层 buffer**：默认会缓冲响应，导致 SSE 流到前端时是"一次性吐"而非"逐字流"。Nginx 要 `proxy_buffering off`；Node 中间层 (Express/Koa) 用 `res.flushHeaders()` + 关闭压缩中间件。
> ② **超长连接超时**：SSE 是长连接，主系统侧 LB / Nginx / Node 框架默认超时（30/60s）会主动断连。必须显式调大到与 RAG 端最长生成时间一致（我们设的是 120s）。
> ③ **心跳保活**：长时间无数据时中间设备会断连。每 15s 发一个 `: heartbeat\n\n` 注释行，前端忽略，链路保活。

**加分项**：
- 主动提到"压缩中间件"——这是常被忽略的 SSE 杀手
- 给出心跳的具体格式（注释行而不是 event）

**避雷点**：
- 别说 "SSE 没什么坑" ——一定有
- 别推荐改成 WebSocket——主系统已用 SSE 就配合，不要为技术偏好破坏架构

### 7.2 流式契约：SSE 事件结构应该怎么约定？（L2）

**意图** —— 验证你能不能把"流式"这种异步交互设计成稳定契约。

**答题骨架**：
> 我们和主系统约定的事件结构（被动接受但参与了讨论）：
> - `event: token` — 一个生成 token，`data` 是字符串片段
> - `event: meta` — 检索到的引用 chunk 元数据（chunk_id / source_path / score），通常在生成开始前一次性下发
> - `event: error` — 结构化错误体（含 code/message/retryable）
> - `event: done` — 流结束信号，`data` 含本次完整统计（总 token 数、耗时分段）
>
> **关键约定**：
> ① 每个事件都有 `id` 字段，方便客户端用 `Last-Event-ID` 重连续传（虽然当前没实现重连，但协议预留）；
> ② 错误事件不要在 HTTP 状态码层面返错——一旦 SSE 流开始，HTTP 200 已发，错误必须以事件形式 inline 在流里；
> ③ `done` 事件必须发——前端依赖它判断是否结束并切换 UI 状态。

**加分项**：
- 区分 `token`/`meta`/`done` 三类事件——体现协议设计意识
- 强调"流开始后错误必须 inline"——这是 SSE 最容易翻车的点
- 提到 `id` 为重连预留

**避雷点**：
- 别只发 token——前端拿不到引用信息
- 别忘 `done` 事件——前端可能一直转圈

### 7.3 上下文透传：哪些字段必须从主系统透传到 RAG？（L2）

**意图** —— 验证你理解分布式系统的可观测性基础。

**答题骨架**：
> 最少必须透传四类：
> ① **trace_id / request_id** — 否则跨服务排查根本没法关联日志；
> ② **user_id / session_id** — 限流/计费/审计都需要；
> ③ **业务标签**（KB 类型、对话场景）— 用于分流、监控分维度统计；
> ④ **租户/团队标识**（如有多租户）— 数据隔离前提。
>
> **怎么透传**：HTTP header 里走（约定 `X-Trace-Id`、`X-User-Id`、`X-KB-Type` 等），FastAPI 用 middleware 统一塞到 request context，业务代码取出来用。**不要让业务代码自己去 header 里翻**——容易漏。

**加分项**：
- 主动提到"middleware 统一处理"——体现工程模式
- 提到 trace_id 是"跨服务排查"的前提——体现分布式视角

**避雷点**：
- 别只说 user_id——trace_id 是更优先的
- 别说"放 body 里"——header 更标准且不污染业务字段

### 7.4 鉴权与边界：RAG 服务如何防止被越权调用？（L3）

**意图** —— 验证你的安全意识。

**答题骨架**：
> RAG 服务**只信任主系统调用**，不直接暴露给前端或外部。两道防线：
> ① **网络层**：部署在内网，对外不开端口；或在 K8s 里只允许主系统 namespace 访问。
> ② **应用层**：主系统调用时带签名 token（HMAC + 时间戳），RAG 验签后才处理。即使内网被穿透，没有签名也调不通。
>
> **数据层防越权**：所有查询必须带 `kb_id` + `user_id`，服务端校验该 user 是否有此 kb 的访问权限——这一步在主系统已做，RAG 仍要做一遍（**纵深防御**），防止主系统某个分支漏掉权限校验时整个系统暴露。

**加分项**：
- 提到"纵深防御"概念——体现系统化的安全思维
- 区分网络层/应用层/数据层三层防护
- 强调"主系统已做仍要再做"

**避雷点**：
- 别说"内网就安全"——是分层防护中的一环不是全部
- 别说"主系统会校验我们就不用"——单点失败会爆炸

### 7.5 错误码与降级：怎么和主系统约定错误体？（L2）

**意图** —— 验证你设计错误处理是否考虑下游的降级决策。

**答题骨架**：
> 与主系统约定的结构化错误体：
> ```json
> { "code": "RAG_RETRIEVAL_TIMEOUT",
>   "category": "system" | "business" | "ratelimit" | "timeout",
>   "message": "可读描述",
>   "retryable": true | false,
>   "fallback_hint": "可选——建议主系统怎么降级" }
> ```
> 关键设计：
> ① **category 分四类**，主系统据此决定 UI——`business` 类直接展示 message 给用户、`system` 类展示 "系统繁忙"、`ratelimit` 类提示"请稍后"、`timeout` 类决定是否重试或降级；
> ② **retryable 字段**让主系统不用猜——5xx 不一定能重试（如幂等性问题）；
> ③ **fallback_hint** 给主系统建议（如 "可以走纯 LLM 兜底"），但不强制。

**加分项**：
- 错误体设计考虑了主系统的"决策需求"而不只是"事实陈述"
- 主动区分 retryable，体现你理解重试的复杂性

**避雷点**：
- 别只返 HTTP 5xx——主系统拿到这个判断不出怎么处理
- 别用自由文本错误——主系统没法据此分流

### 7.6 超时分段：RAG 链路里超时该怎么设？（L2）

**意图** —— 验证你能不能识别"长链路单点拖死全链路"的风险。

**答题骨架**：
> RAG 链路至少四段，每段独立超时：
> ① **Embedding API 调用** — 5s 超时，超时不重试（已是退避后）；
> ② **检索（LanceDB + Meilisearch）** — 2s 超时，超时降级为单路召回；
> ③ **LLM 生成** — 90s 总超时，含首 token 5s（首 token 卡住直接放弃）；
> ④ **整体兜底** — 整个请求 120s 硬超时，超时直接关流。
>
> 主系统侧的超时要设在 RAG 整体兜底之上（如 130s），否则主系统先超时关连接，RAG 还在算，资源浪费。

**加分项**：
- 提到"首 token 超时"——这是流式系统的关键指标
- 主动说"主系统超时要设在 RAG 之上"——体现跨服务超时设计意识
- 段超时短于总超时是基本原则

**避雷点**：
- 别只说一个总超时——单段卡住时整个链路都卡
- 别忘"超时后资源清理"——异步任务要 cancel

### 7.7 限流与隔离：怎么避免一个调用方打挂整个 RAG？（L3）

**意图** —— 验证你的多租户/多业务方意识。

**答题骨架**：
> 多层限流 + 隔离：
> ① **全局限流**：Redis 滑窗，按 RAG 服务整体 QPS 设上限（如 100），保护服务不被打挂；
> ② **维度限流**：按 user_id / kb_id / 业务场景分别限流，防止"某一个用户/某一类查询打满全部配额"；
> ③ **资源隔离**：embedding/LLM 调用走独立的 httpx client + 连接池，避免某段慢拖累其他段；
> ④ **降级开关**：配置中心里有"召回增强关闭""动态阈值降级为固定值"等开关，遇压力可立即简化流水线降低负载。

**加分项**：
- 区分"全局限流"和"维度限流"——前者保命、后者保公平
- 提到"降级开关"——体现你考虑过应急预案

**避雷点**：
- 别只说"加限流"——要说限流的颗粒度
- 别忽略"降级"——限流是拒绝服务，降级是降质服务，两者都要

### 7.8 可观测性：至少要记录哪些日志和指标？（L2）

**意图** —— 验证你做过真实的线上排查，知道缺什么会要命。

**答题骨架**：
> **每次请求必记**：
> - trace_id / user_id / kb_id / query 文本
> - 召回 chunk 的 id 列表 + 各路分数（向量分、BM25 分、RRF 分）
> - 各段耗时（embedding / 检索 / 融合 / LLM 首 token / LLM 总耗时）

> - 最终错误码（成功为 0）
>
> **聚合指标**（Prometheus / 日志聚合）：
> - QPS / P50/P95/P99 延迟（按段拆）
> - 错误率（按 category 拆）

**加分项**：

- 强调 trace_id 串通

**避雷点**：
- 别只记 P99——P50/P95/P99 一起看才完整
- 别忘记 "embedding 输入文本" 也要记（debug 时要复现）

### 7.9 版本兼容：接口要怎么演进？（L2）

**意图** —— 验证你理解接口迭代的兼容性原则。

**答题骨架**：
> 即使接口由主系统定，演进也要双方协商。原则：
> ① **新增字段走 optional + 默认值**，不破坏老调用方；
> ② **删除字段必走 deprecate 流程**：① 标记为 deprecated 但仍接受 → ② 监控老用法占比降到 < 1% → ③ 真正下线，至少 1 个月窗口；
> ③ **行为不向后兼容的改动必须升版本号**（URL 路径加 /v2 或 header 带 version），新老并存一段时间；
> ④ **流式协议变更尤其慎重**——前端解析 SSE 事件类型时强类型，加了新事件类型未升级的前端可能崩。

**加分项**：
- 提到"deprecate 三阶段流程"——体现协作意识
- 单独强调"流式协议变更慎重"——这是 SSE 系统的特殊点

**避雷点**：
- 别说"接口被动接受所以不用我管"——演进是双方的
- 别一上来就 v2——能兼容尽量兼容

### 7.10 一次跨团队故障的端到端处理（L3）

> ⚠️ **本题用一个示意性故障，你可以替换成项目中真实经历过的事件**。

**意图** —— 验证你能不能讲清"事故定位 → 根因 → 长期改进"完整闭环。

**答题骨架**（示意）：
> **现象**：某次发布后用户反馈"对话偶发返回错误"，主系统团队报警，怀疑 RAG 服务问题。
>
> **定位过程**（用 trace_id 串四段链路）：
> ① 前端日志 → 主系统 trace_id；
> ② 主系统日志 → 显示调用 RAG 在 30s 后超时；
> ③ RAG 日志 → 同一 trace_id 显示 LLM 调用耗时 35s；
> ④ LLM provider 日志 → 上游 API 当时整体抖动到 P99 = 40s。
>
> **根因**：① 上游 LLM API 偶发抖动；② 主系统侧超时设了 30s（小于 RAG 总超时 120s），主系统先放弃，RAG 仍在跑导致连接异常。
>
> **短期处理**：① 主系统超时调到 130s（与 RAG 兜底对齐）；② RAG 加 LLM 首 token 超时 5s，首 token 卡住即放弃避免无效等待。
>
> **长期改进**：① 全链路 trace_id 强制串通的合规检查；② 各服务超时层级文档化（主系统 ≥ RAG 总超时 ≥ 单段超时）；③ 上游 LLM 抖动监控告警，提前切到备用 provider。

**加分项**：
- 用 trace_id 串链路定位——展示真实的排查能力
- 区分"短期止血"和"长期改进"——体现工程节奏感
- 长期改进里有"机制层面"的改动（合规检查 + 文档化），不止当前问题

**避雷点**：
- 别甩锅 LLM provider——要展示"我们能在抖动下做什么"
- 别只讲短期 fix——要展示长期视角

**待补充真实素材**：项目里如果有真实跨团队故障，按这个结构改写更有说服力。

### 8.2 最大的失败是什么？（L2）

**意图** —— 验证你的反思能力 + 心理成熟度。

**答题骨架**：

### 8.3 主动暴露不足时怎么把握尺度？（L2）

**意图** —— 验证你的自我认知准确度。

> ① **暴露的不足必须配套"我知道下一步怎么做"**——否则就是裸露弱点而不是体现成长；
