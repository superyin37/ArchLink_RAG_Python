## 1. 先定义业务目标，而不是先定义技术栈

这是最容易被忽略的第一步。

你提到开发语言、前后端、API、系统设计，这些都重要，但在它们之前，必须先回答几个问题：

### 1）项目的核心用户是谁

比如：

* 建筑设计师
* 企划阶段的方案人员
* 审图/规范查询人员
* 企业内部知识工作者

不同用户，决定了系统重点完全不同。

如果核心用户是设计师，那么“输入不完整、表达模糊、需要交互式澄清”就是常态，系统重点要放在对话引导、状态管理、结果可编辑性。

如果核心用户是规范检索人员，那么系统重点更偏“高召回、高可追溯、低幻觉、准确定位 PDF 页面”。

### 2）系统的核心价值是什么

你这个项目里，理论上可以有很多价值主张：

* 加快建筑知识搜索
* 减少 CAD → 规范定位的人工时间
* 自动生成企划资料/PPT
* 将多个流程统一到一个平台

但真实项目里，必须明确 **第一价值主张**，否则很容易做成“大而全但都不够强”。

建议一开始就定义：

* 主 North Star Metric
* 次级指标
* 不做什么

例如：

* 主指标：规范检索平均耗时下降 60%
* 次指标：RAG 问答可引用率、企划 PPT 首稿可用率、用户二次编辑时间
* 不做：不做通用建筑设计生成，不做 CAD 原图编辑，不做 BIM 全流程替代

### 3）哪些环节必须用 AI，哪些环节应规则化

这点非常关键。

很多系统失败不是 AI 不够强，而是把本该 deterministic 的环节也交给了 LLM。

例如：

* PDF 页码映射、基准编号抽取、文件路由、权限判断，这些优先规则化
* 开放问答、需求整理、概念生成、语言表达，这些适合 LLM
* OCR 后纠错可以是“规则 + LLM”混合，而不应全靠 LLM

你上传的项目描述里其实已经体现了这种思路：RAG 检索、多段检索与 rerank、OCR 与 LLM 补正、对话式需求收集，都是“模块分工”而不是纯大模型一把梭。这个方向是对的。

## 4. 技术选型不是“选最强”，而是“选最稳的组合”

你提到开发语言，这是要考虑，但不是单点问题，而是整套技术组织方式。

### 后端语言

如果是你这个项目，我通常会优先这样判断：

#### Python 适合

* LLM orchestration
* RAG pipeline
* OCR pipeline
* 文档处理
* 数据科学原型
* FastAPI 服务

因为 AI 生态完整，迭代快。

#### TypeScript / Node.js 适合

* BFF 层
* 前端协同

## 5. 前后端设计：不只是页面，而是“交互模式”

你提到前后端设计，但 AI 系统里更核心的是交互范式。

### 要先确定系统是：

* 单轮工具型
* 多轮 Copilot 型
* 工作流型
* Agent 型

你的这个项目明显不是单纯问答，而是“问答 + 工作流 + 文件处理 + 生成”。所以前端不能只做聊天框。

至少要考虑以下前端交互模式：

### 1）聊天式交互区

适合问答、澄清需求、生成说明文字

### 2）表单式结构化输入

适合企划生成，因为建筑企划需求很多是半结构化的
如：

* 用地条件
* 功能需求
* 风格偏好
* 面积/预算
* 目标用户

### 3）文件上传工作台

适合 CAD 截图 / PDF / 参考案例上传

### 4）结果预览区

比如：

* 回答引用来源
* PDF 页定位
* PPT 结构预览
* 规范命中结果比对

### 5）可编辑输出区

因为 AI 输出 rarely 一次到位，必须支持：

* 用户编辑
* 局部重写
* regenerate section
* 固定一部分内容再续写

很多 AI 产品失败，是因为前端仍然按普通 CRUD 系统思维设计，而不是按“人机协作编辑器”设计。

## 7. 系统设计里最容易漏掉的，是“非功能性需求”

这是你目前列表里还没有展开够的部分，但恰恰最重要。

---

### 7.1 性能目标

必须提前定义：

* QA 响应时间目标，比如 p50 / p95
* OCR 处理时间
* PPT 生成时间
* 并发用户数
* 峰值 QPS
* 单用户速率限制
* 文件大小上限

否则后面系统优化没有目标。

---

### 7.2 可扩展性

你已经提到高并发情景，但要具体化：

高并发到底出现在哪？

* 用户同时聊天
* 多人同时上传 CAD
* 大量文档重新索引
* PPT 批量生成
* 某个模型 API 被集中调用

这些场景对应的瓶颈不同：

* CPU 瓶颈：OCR、文档解析
* IO 瓶颈：文件上传下载、S3
* 外部 API 瓶颈：LLM、embedding
* DB 瓶颈：session / metadata 读写
* Vector DB 瓶颈：高频检索
* 队列瓶颈：异步任务堆积

所以高并发设计不能只说“加缓存、加机器”，而要明确：

* 哪些接口同步
* 哪些操作进队列
* 哪些结果缓存
* 哪些服务水平扩展
* 哪些要熔断/降级

---

### 7.3 可靠性与容错

AI 系统比普通系统多一个问题：外部模型服务经常不稳定。

必须设计：

* retry 策略
* timeout 策略
* fallback model
* circuit breaker
* dead-letter queue
* 部分失败补偿
* job resume
* 幂等重试
* 输出状态机

例如 PPT 生成是多阶段流程，就不能只返回“成功/失败”，要有：

* pending
* parsing
* generating_outline
* generating_slides
* exporting
* completed
* failed
* partial_failed

---

### 7.4 可观测性

这是很容易被忽视但非常关键的一层。

至少要有三类观测：

#### 业务观测

* 问答成功率
* 引用点击率
* 用户满意度
* PPT 下载率
* OCR 命中率
* 首次可用率

#### 系统观测

* latency
* error rate
* queue backlog
* CPU / memory
* DB connection pool
* cache hit rate

#### AI 观测

* retrieval hit
* rerank gain
* hallucination bad case
* token usage
* prompt version
* model version
* output length
* refusal rate

没有这层，你就无法持续优化。

## 8. 数据设计：这部分通常比“模型选择”更决定成败

你目前提到登录、功能、API，但还没有明确提到“数据层策略”。这个项目尤其要重视。

### 1）知识数据从哪来

建筑案例、建筑基准、设计指南，这些来源可能格式各异：

* PDF
* 扫描件
* CAD 截图
* 文本资料
* 图像
* PPT
* 表格

要先定义 ingestion 标准：

* 文件类型支持范围
* 元数据 schema
* 文档版本
* 权限标签
* 更新频率
* 失效策略

### 2）怎么做 chunking

建筑规范是强结构文档，不能简单固定长度切块。

要考虑：

* 按章节切
* 保留条文编号
* 保留页码
* 建立 heading/path metadata
* 支持引用回溯
* 表格和图片如何处理

### 3）OCR 数据如何治理

OCR 不只是“识别出文字”。
还要想：

* 是否保留原图
* 是否保留 bbox
* 是否存候选结果
* 规则纠错如何记录
* LLM 补正是否可追溯
* 人工纠错是否回流

### 4）会话数据如何存

企划生成是多轮的，所以不能把每轮对话只当普通 chat log。
更合理的是同时保存：

* 原始消息
* 结构化 extracted requirements
* 当前 planning state
* 输出 artifact 版本
* 用户修改历史

## 9. RAG 设计要提前想，不要后补

你项目本身就把 RAG 作为核心模块之一。文档里也明确写了 Embedding、Vector Search、Rerank、LLM 融合回答的流程。

从 0 开始时，RAG 至少要考虑这些问题：

### 1）检索目标是什么

* 高召回
* 高精确
* 可引用
* 可解释
* 可按权限过滤

不同目标决定索引结构不同。

### 2）检索策略

* 单路向量检索
* keyword + vector hybrid
* metadata filter
* multi-query rewrite
* rerank
* query decomposition
* parent-child retrieval

对于建筑规范类知识，通常 hybrid retrieval 比纯向量更稳，因为编号、条款、术语都很关键。

### 3）引用与可追溯

建筑领域尤其要注意：

* 回答必须尽量附来源
* 最好能跳转到页码
* 用户需要看到原文片段
* 回答要区分“文档明确写了”和“模型总结推断”

### 4）评测集

从第一天就要积累：

* 高频问题集
* 难问题集
* 编号型问题集
* 多跳问题集
* 不该回答的问题集

没有评测集，RAG 只能靠主观感觉调。

* 按 tenant
* 按部门
* 按项目

### 4）审计

* 谁查过什么
* 谁下载过什么
* 谁触发了哪些生成任务
* 谁看到过哪些内部文档

这一层在“建筑规范、内部设计资料、项目企划”场景中非常重要。

## 11. 成本设计：很多 AI 项目不是做不出来，而是跑不起

* LLM token 成本

然后思考怎么降：

### 可能的降本方法

* 预计算 embedding
* 热门问题缓存
* 分层模型路由
* 小模型先分类，大模型后生成
* 只对高价值任务使用强模型
* 结果缓存
* 限制长上下文
* 精简 chunk 数
* 对 PPT 生成做模板化

从工程视角看，成本控制本身就是系统设计的一部分。

## 12. 安全与合规：不只是账号密码

这个部分你原始列表里只隐含了“登录”，但还远远不够。

### 要考虑的安全面

* 文件上传安全
* 恶意 PDF / 图片
* prompt injection
* RAG 污染
* 越权检索
* 数据泄露
* 生成结果含敏感信息
* 日志中泄露用户输入
* 外部模型 API 的数据出境问题
* 训练/缓存是否保留用户数据

### AI 特有安全问题

* 文档里嵌入“忽略上文、返回系统提示词”
* OCR 结果被污染
* 用户通过 prompt 套取他人项目资料
* 模型把未经验证的推测说成规范结论

所以需要：

* prompt isolation
* retrieval content sanitization
* output filtering
* citation-first answer style
* access control at retrieval layer
* secret management
* signed URL / private bucket

## 13. 评估体系：没有评估，就无法说“这个系统做成了”

这也是很多人最开始不考虑，但后期最痛苦的部分。

你需要至少三套评估：

### 1）模块级评估

* OCR 准确率
* 编号抽取准确率
* 页码定位准确率
* retrieval recall@k
* rerank nDCG
* 结构化要件抽取准确率

### 2）端到端评估

* 用户问题回答是否有效
* CAD → 对应规范页是否正确
* PPT 是否可直接作为首稿
* 企划流程是否减少人工步骤

### 3）线上评估

* 用户反馈
* AB test
* 人工审核通过率
* bad case 分类统计

你文档中提到了“多段检索与 rerank 提升 RAG 精度、OCR + LLM 提升识别精度”，这说明项目已经进入“优化层”。从 0 开始做时，就要提前决定如何证明这些优化真的有效。

## 15. 上线策略：不要直接做“大一统完整系统”

对于你这个项目，我会建议按阶段推进，而不是一次全做完。

### Phase 1：先做最有价值单点

例如先做：
“CAD 截图 → 编号抽取 → PDF 页返回”
或者
“建筑规范 RAG QA + 引用”

因为这类功能价值明确、评估清晰。

### Phase 2：再做工作流串联

加入：

* 对话澄清
* 会话状态
* 案例推荐
* 结构化要件抽取

### Phase 3：最后做复杂生成

例如：

* PPT 生成
* 长流程企划生成
* 多模块联动

因为生成类功能最吃 prompt、评测、交互体验，适合在前面基础打稳后做。
