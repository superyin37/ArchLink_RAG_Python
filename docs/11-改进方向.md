# 11 - 改进方向（Roadmap）

> 本文盘点当前 Python 版 RAG 系统在**检索质量、模型与推理、工程与可扩展性、可观测与评测、安全合规、产品体验、生态集成**等维度上可演进的方向。目的是作为路线图候选池，而非立即执行清单。条目尽量穷举，排期时再按 ROI 取舍。

---

## 0. 总览

按改造层次可分为四层：

1. **数据层**：解析、切片、嵌入、索引、存储
2. **检索层**：查询改写、召回、融合、重排、上下文组装
3. **生成层**：LLM 路由、缓存、工具调用、结构化输出、评测
4. **平台层**：可观测、可扩展、可运维、安全、多租户、成本

下文按"方向 → 现状 → 痛点 → 目标 → 实施要点 → 优先级"展开。优先级分为 **P0（快赢）/ P1（中期）/ P2（战略）**。

---

## 1. 文档解析与预处理

### 1.1 PDF 结构化解析
- **现状**：`loaders/pdf.py` 使用 `PyPDF2` / `pdfplumber`，仅抽取纯文本，版式/表格/图片丢失
- **痛点**：扫描件、双栏论文、复杂表格、公式几乎不可用
- **目标**：保留标题层级、表格单元、图片标注、公式 LaTeX
- **方案**：
  - 引入 `unstructured` / `Marker` / `PP-StructureV2` / `MinerU`
  - 为扫描件接入 OCR（PaddleOCR / Tesseract / PP-OCRv4）
  - 表格→Markdown，公式→LaTeX，图片→描述 + 引用
- **优先级**：P1

### 1.2 DOCX 解析增强
- **现状**：`python-docx` + `mammoth`，重点是标题树
- **痛点**：批注、修订历史、图文混排、目录、页眉页脚、交叉引用未处理
- **方案**：保留批注为 meta；忽略目录块；解析 SmartArt 为可描述的文本
- **优先级**：P1

### 1.3 Markdown 方言扩展
- **现状**：`parse_markdown_optimized` 处理标准语法
- **痛点**：GFM 表格、脚注、数学公式、Mermaid、front-matter、嵌入式 HTML 未专门处理
- **方案**：接入 `markdown-it-py` + 插件，统一 AST；front-matter 抽取成 `RagDocument.meta`；Mermaid 保留原文并打标签
- **优先级**：P0

### 1.4 新增格式支持
- PowerPoint（.pptx）：`python-pptx`，逐页解析，把 slide 当一级节点
- HTML / 网页抓取：`trafilatura` / `readability-lxml` 去噪后进入 Markdown 流水线
- CSV / TSV：每行一 chunk，可配置按主键聚合
- 代码仓库：按语法树（tree-sitter）切片，保留函数/类粒度
- 图片/截图：OCR + 视觉描述（VLM 如 GPT-4o / Qwen-VL / GLM-4V）
- 音视频：Whisper 转录 + 说话人分离 + 时间戳 chunk
- **优先级**：P1 / P2（按业务需要）

### 1.5 智能切片（Chunking）
- **现状**：基于标题树 + 段落合并 + 大小切分
- **痛点**：语义边界被标题粒度绑死；长表格/长代码被硬切
- **方案**：
  - **Semantic Chunking**：相邻句子 embedding 余弦距离突变处切分
  - **Proposition-based Chunking**：LLM 抽取为原子命题
  - **Late Chunking**：先对整文档用长上下文 embedder 编码，再在 token 维度切片，保留全局语义
  - **Layout-aware Chunking**：PDF/DOCX 按段/表/图分别切，表格整块不拆
  - **自适应粒度**：同一 KB 支持多粒度索引（句/段/节），检索时再选层
- **优先级**：P1

### 1.6 元数据富化
- 自动抽取：作者、时间、标签、摘要、关键词、命名实体
- 文档级摘要与节级摘要同时入索引（Small-to-Big、Parent-Document Retrieval 基础）
- 语言检测 + 领域分类，辅助后续过滤
- **优先级**：P1

## 2. 嵌入与向量化

### 2.1 嵌入层工厂化
- **现状**：`embedding/doubao.py` 单一实现，HTTP 直连
- **痛点**：切换模型成本高；无法按 KB 选嵌入
- **方案**：参照 `LLMOne` 建 `EmbeddingOne` 工厂，`openai / bge / voyage / jina / cohere / gemini` 可插拔；`KnowledgeBase.embedding_provider_id` 决定使用者
- **优先级**：P0

### 2.2 批处理与限流
- **现状**：逐批调用 Doubao，重试与批大小由 loader 自行决定
- **方案**：统一 `batch_embed(texts, batch_size, concurrency)`，自适应缩小 batch 应对 429；按 Token 而非条数装箱；断点续传（chunk 状态位）
- **优先级**：P0

### 2.3 多维度混合嵌入
- 稀疏向量（SPLADE / BM25 向量）+ 稠密向量双索引，支持 **Hybrid Dense+Sparse**
- 多语言嵌入（`bge-m3`）支持中英混合检索
- **优先级**：P1

### 2.4 向量量化与压缩
- Matryoshka Embedding（可截断维度）：存储可按需压缩
- PQ / SQ / Binary 量化减少内存与 I/O
- **优先级**：P2

### 2.5 嵌入模型版本治理
- **痛点**：换嵌入模型必须全量重算
- **方案**：索引版本化（`kb_{id}_v{n}`），双写灰度切换，`SEARCH` 按当前生效版本读
- **优先级**：P1

## 3. 索引与存储

### 3.1 向量存储抽象
- **现状**：代码直接依赖 LanceDB
- **目标**：定义 `VectorStore` 协议（`upsert / search / delete / create_index`），让 LanceDB / Qdrant / Milvus / pgvector / Weaviate / Elasticsearch-kNN 可替换
- **收益**：水平扩展、云托管、HA
- **优先级**：P1

### 3.2 全文搜索增强
- **现状**：Meilisearch，中文默认分词一般
- **方案**：
  - 评估 Elasticsearch/OpenSearch + IK/HanLP 分词
  - 或在 Meilisearch 中配置停用词、同义词、Ranking Rules
  - 引入 BM25F（字段加权：heading > content > code）
- **优先级**：P1

### 3.3 元数据过滤与混合查询
- 现状：检索按 KB 隔离，文档维度过滤能力有限
- 方案：支持任意 meta 字段（tag、date range、author、lang）做 **pre-filter** 或 **post-filter**；LanceDB/Qdrant 都原生支持谓词
- **优先级**：P0

### 3.4 关系型数据库优化
- 大表（chunks / call_log）分区或归档；软删只保留 90 天硬删清理 job
- 读写分离（MySQL 从库），检索路径走只读
- JSON meta 列加函数索引或改 GEN COLUMN
- **优先级**：P1

### 3.5 文件存储迁移
- **现状**：本地 `uploads/` 挂载卷
- **方案**：对接 S3 / MinIO / OSS，`FileStore` 抽象；支持预签名 URL 直传，减少 app I/O
- **优先级**：P1

## 4. 检索与排序

### 4.1 查询理解与改写
- **Multi-Query**：LLM 生成 N 个等价问法并行检索后合并
- **HyDE**：LLM 先生成假设答案，用其 embedding 检索
- **Step-back Prompting**：抽象出上位问题再检索
- **查询分类**：事实型/推理型/闲聊型路由不同管线
- **历史感知改写**：基于对话上下文把"它怎么用"改写成独立查询
- **优先级**：P0

### 4.2 召回策略
- **稀疏+稠密混合**：BM25 + Dense + ColBERT late-interaction
- **Small-to-Big**：以小 chunk 命中后回取父节点供生成，兼顾精度与上下文
- **Parent-Document Retriever**：命中子句回取所属段落
- **MultiVector**：一个 chunk 多个表示（摘要、问答、关键词）
- **Ensemble Retriever**：多召回源按权重合并
- **优先级**：P0 / P1

### 4.3 重排（Rerank）
- **现状**：`readability.py` 启发式
- **方案**：
  - 接入 Cross-Encoder（`bge-reranker-v2-m3` / Cohere Rerank / Jina Rerank）
  - LLM-as-Reranker（小模型相关性评分）
  - Listwise Rerank（RankGPT）
- **优先级**：P0

### 4.4 融合策略进阶
- 现状：RRF / weighted / linear
- 新增：分数归一化（min-max、z-score）；DBSF；学习到排（LambdaMART）
- 多路融合支持 > 2 路（向量、BM25、稀疏、图、SQL 过滤）
- **优先级**：P1

### 4.5 上下文组装
- 当前按 Token 预算 + 树组装
- 新增：**Lost-in-the-Middle 缓解**（关键 chunk 放首尾）；**LLMLingua 压缩**（对长上下文做 prompt 压缩）；**引用锚点**（每段附 chunk_id，便于生成端带引用）
- **优先级**：P1

### 4.6 自适应检索
- 判断是否需要检索（self-RAG / FLARE），闲聊跳过向量库
- 判断是否需要多轮检索（agentic retrieval）
- 阈值自适应当前基于分数分布，可加入"覆盖度"指标（命中文档多样性）
- **优先级**：P1

### 4.7 跨 KB 联合检索
- 多 KB 路由（Router LLM 决定查哪几个）
- 加权聚合 + 去重
- **优先级**：P1

## 5. LLM 与生成

### 5.1 Provider 完整性
- 补齐 Bedrock / Vertex AI / Mistral / DeepSeek-R1 / Qwen 全家 / 本地 vLLM / Ollama
- 统一流式事件协议（content/tool_call/thinking/usage），兼容 Claude 的 thinking blocks 与 OpenAI o-系列 reasoning
- **优先级**：P1

### 5.2 工具调用统一抽象
- 当前 `completions` / `responses` 两套路径
- 新增 `ToolRegistry`：JSON Schema 定义 → 多 provider 自动转换（OpenAI function / Anthropic tools / Gemini function declaration）
- 支持并行工具调用、工具超时、安全沙箱
- **优先级**：P1

### 5.3 结构化输出
- outline / ppt 阶段依赖 JSON，遇解析失败易中断
- 方案：`instructor` / `outlines` / JSON Schema + 自修复重试；Pydantic v2 校验；失败降级到 text + 后处理
- **优先级**：P0

### 5.4 缓存体系
- **现状**：Anthropic prompt cache、OpenAI 自动缓存
- 新增：
  - **语义缓存**：按 query embedding 命中历史回答（阈值 + TTL）
  - **检索结果缓存**：同 query + 同 KB 版本命中即返回
  - **LLM 响应缓存**：稳定 prompt（system、tools）切片缓存
  - **前缀缓存**：与 vLLM / SGLang 结合部署本地模型
- **优先级**：P0

### 5.5 成本与路由
- 当前 fallback 仅按失败触发
- 新增：按 **延迟 / 价格 / 上下文长度 / 任务类型** 动态路由；复杂任务用强模型，闲聊用便宜模型；预算约束（用户/租户级 token 上限）
- **优先级**：P1

### 5.6 Prompt 工程化
- 当前 `prompt.py` 按阶段拼装
- 建议：
  - **Prompt 模板中心**（Jinja2 / mustache），版本化
  - **Prompt 注册表**：阶段→模板→模型能力映射
  - **自动评测与回归**（prompt 变更触发评测）
  - **运行时变量注入审计**（防 prompt injection）
- **优先级**：P1

### 5.7 Agentic 能力
- 现有 PPT 阶段机 → 扩展为通用 Workflow Engine（LangGraph / 自研状态图）
- 引入 Planner / Executor / Critic 三角
- 支持长任务 checkpoint + 断点恢复
- **优先级**：P2

## 6. 可观测性与评测

### 6.1 分布式追踪
- OpenTelemetry 贯穿 router → service → 外部调用
- 一次请求的 trace：Auth → Retrieval Pipeline（每一级 span）→ LLM 调用（含 provider、model、tokens、TTFT）
- 导出到 Jaeger / Tempo / Honeycomb
- **优先级**：P0

### 6.2 指标与告警
- Prometheus 指标：QPS、P50/P95/P99 延迟、TTFT、错误率、命中率、recall@k、上下文长度分布、token 消耗
- Grafana 面板：按 KB / 用户 / 模型维度
- 告警：嵌入 API 失败率、LLM provider 错误、检索延迟恶化
- **优先级**：P0

### 6.3 日志规范
- 结构化日志（JSON），字段固定（trace_id、user_id、kb_id、stage）
- 日志分级分流：业务 / 审计 / 性能
- 调用日志表（已有）与 ELK/Loki 互补
- **优先级**：P1

### 6.4 离线评测（Eval Harness）
- **数据集构建**：人工标注 query→(gold docs, gold answer) 金标集；从真实 query 抽样脱敏
- **RAGAS 指标**：faithfulness、answer_relevance、context_precision、context_recall
- **DeepEval / TruLens / promptfoo** 一键回归
- **CI 门禁**：prompt/模型/检索参数变更触发评测，低于阈值阻止合入
- **优先级**：P0

### 6.5 在线反馈闭环
- 前端增加 👍/👎/改写/引用点击埋点
- 反馈写入 `message.meta` 与 `call_log`
- 用于：rerank 模型再训、bad case 挖掘、prompt 回归集
- **优先级**：P1

### 6.6 Bad Case 工作台
- 管理后台查询某条 trace：原 query → 召回列表（含分数）→ rerank → 上下文 → 最终 prompt → LLM 输出
- 一键复现、一键加入评测集
- **优先级**：P1

## 7. 安全与合规

### 7.1 认证授权
- 当前 JWT + 白名单
- 新增：RBAC（role-permission）、API Key 机制、OAuth2/OIDC 集成、MFA
- 租户隔离：`tenant_id` 贯穿所有表，查询强制注入
- KB 级 ACL（读/写/管理），文档级敏感度标签
- **优先级**：P1

### 7.2 Prompt Injection 与越狱防护
- 输入侧：检测可疑指令（"忽略之前指令"等模式）
- 检索侧：把 retrieved content 明确标记为"参考资料"而非指令
- 输出侧：敏感响应过滤（PII、暴力、政治）
- 接入 **Llama Guard / NeMo Guardrails / ShieldGemma**
- **优先级**：P0

### 7.3 数据安全
- 上传文件 MIME 校验 + 病毒扫描（ClamAV）
- 存储加密（MySQL TDE / S3 SSE / LanceDB at-rest）
- 传输加密强制 HTTPS，内部服务 mTLS
- 密钥管理用 Vault / AWS KMS，不留在 .env
- **优先级**：P0

### 7.4 隐私与合规
- PII 检测与脱敏（`presidio`）
- 数据留存策略（按租户/法规配置）
- GDPR "被遗忘权"：删除用户数据需级联清空向量库与日志
- 审计日志：谁何时查了哪些文档，谁训练用了哪些数据
- **优先级**：P1

### 7.5 速率与配额
- 当前 Redis 限流
- 扩展到：租户配额、KB 配额、每模型 TPM/RPM、费用配额
- 超限友好降级（返回缓存 / 排队）
- **优先级**：P1

## 8. 可扩展性与部署

### 8.1 无状态化
- 把本地 `uploads/` 改走对象存储，app 可水平扩容
- LanceDB 若保留则走共享存储（NFS / EFS）或迁到 Qdrant/Milvus 集群
- Session 走 Redis 而非内存
- **优先级**：P1

### 8.2 异步任务队列
- **现状**：FastAPI BackgroundTasks（进程内、重启丢失）
- **方案**：ARQ / Dramatiq / Celery + Redis / RabbitMQ；重试、优先级队列、死信队列、可视化（Flower）
- 文档处理、嵌入、索引、评测均迁入队列
- **优先级**：P0

### 8.3 容器编排
- Kubernetes Helm Chart / Kustomize
- HPA 基于 QPS 或队列深度
- 滚动发布、蓝绿/金丝雀
- **优先级**：P1

### 8.4 多环境与配置
- dev / staging / prod 分环境 Chart
- 配置中心（Nacos / Consul）热更新阈值参数，避免改一个阈值就发版
- Feature Flags（Unleash / GrowthBook）做功能开关与 A/B
- **优先级**：P1

### 8.5 CI/CD
- pytest + coverage；typing（mypy / pyright）；lint（ruff）；安全扫描（bandit、pip-audit、trivy）
- pre-commit hooks
- GitHub Actions / GitLab CI 流水线；镜像签名（cosign）
- **优先级**：P0

### 8.6 依赖治理
- `numpy` 隐式依赖显化
- 锁文件（`uv.lock` / `poetry.lock`）
- Renovate / Dependabot 自动升级 + 回归评测门禁
- **优先级**：P0

## 9. 成本优化

- **Token 审计**：实时看板 + 按用户/KB/租户成本
- **模型降级策略**：预算紧张自动切便宜模型
- **批处理 API**：OpenAI / Anthropic 批量 API 50% 折扣用于离线任务（文档摘要、评测）
- **本地模型托管**：高频便宜任务走 vLLM/Ollama（摘要、分类、Rerank）
- **嵌入去重与复用**：同文本只嵌入一次（内容哈希缓存）
- **冷数据归档**：低访问 KB 向量下沉到便宜存储
- **优先级**：P1

### 10.3 音视频
- Whisper 转录 + 说话人分离；时间戳级 chunk
- 回答带"跳转到 00:12:35"能力
- **优先级**：P2

### 11.3 本体与 schema
- 领域本体建模（业务概念、同义词、上下位关系）
- 支持 schema-aware 检索与生成
- **优先级**：P2

## 12. 产品与体验

### 12.1 前端
- 当前单 HTML + Vue 3 CDN
- 升级：Vite + TypeScript 工程化，组件化；SSR/SSG 用 Nuxt
- UX：流式打字、引用气泡、检索过程可视化、Markdown 渲染完善（代码复制、Mermaid、数学）
- **优先级**：P1

### 12.2 引用与可追溯
- 回答中内嵌 `[1][2]` 引用，点击跳转到原文位置（chunk_id + offset）
- 高亮命中句
- 展示置信度（基于 rerank 分数）

| 类别 | P0（3 个月） | P1（6–12 个月） | P2（12 个月+） |
|---|---|---|---|

## 17. 执行建议

1. **先平台后算法**：可观测+评测+队列先立起来，否则后续所有算法改进都无从度量
2. **每改一处给指标**：任何算法变更必须伴随 RAGAS / 在线指标对比
3. **抽象优先**：VectorStore / EmbeddingOne / FileStore / ToolRegistry 四个抽象是后续演进的地基，尽早沉淀
4. **小步灰度**：语义缓存、重排、查询改写都可做成 feature flag，按 KB / 用户灰度
5. **避免过度设计**：GraphRAG / Agentic 属于战略储备，不要在平台层没打牢前投入
