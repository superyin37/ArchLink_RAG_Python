# 多语言支持开发文档（i18n）

> 状态：规划中
> 创建日期：2026-03-10
> 相关文件：`static/test-ui/index.html`、`app/modules/ppt/`、`app/modules/rag/`

---

## 一、功能范围

| 功能 | 说明 |
|------|------|
| 前端 UI 多语言 | 界面文字支持中/英切换，`localStorage` 持久化 |
| LLM 对话语言匹配 | System prompt 语言自动跟随前端当前语言 |
| RAG Query 翻译 | 非中文 query 先翻译为中文再检索，知识库内容不变 |

**不在范围内**：
- 知识库文档重新 embedding（RAG 数据层保持中文不动）
- 翻译 API 对接（翻译通过现有 LLM 完成，无需第三方翻译服务）

---

## 二、整体方案

### 2.1 前端语言切换

```
用户切换语言 → localStorage 持久化 → 所有 UI 文字切换
                                   → 后续请求携带 X-Language 头
```

### 2.2 RAG 对话流程（跨语言）

```
用户输入（英文）
      │
      ▼
[后端] 检测 X-Language != zh-CN
      │
      ▼
[小模型] 翻译 query → 中文
      │
      ▼
RAG 检索（中文 chunks）
      │
      ├── chunks 原样传给 LLM（不翻译）
      │
      ▼
[LLM] System prompt 末尾追加："Please respond in English."
      + 中文 context chunks
      + 用户原始英文 query
      │
      ▼
流式输出英文回答
```

> **核心原则**：只翻译 query（一次），LLM 原生支持"读中文、答英文"，无需翻译 chunks。

### 3.3 状态与方法

```javascript
// 状态
const currentLang = ref(localStorage.getItem('lang') || 'zh-CN')
const t = computed(() => LOCALES[currentLang.value])

// 切换
function toggleLang() {
  currentLang.value = currentLang.value === 'zh-CN' ? 'en-US' : 'zh-CN'
  localStorage.setItem('lang', currentLang.value)
}
```

### 3.4 请求头注入

所有 `fetch` 请求加入语言头，后端据此决定是否翻译 query：

```javascript
function buildHeaders() {
  const headers = { 'Content-Type': 'application/json' }
  if (authToken.value) headers['token'] = authToken.value
  headers['X-Language'] = currentLang.value   // 新增
  return headers
}
```

## 四、后端实现

### 4.1 语言上下文提取

在 `app/middleware/` 或请求入口处提取语言：

**文件**：`app/middleware/context.py`（或现有中间件）

```python
from starlette.middleware.base import BaseHTTPMiddleware

class LanguageMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request: Request, call_next):
        lang = request.headers.get("X-Language", "zh-CN")
        # 存入 request.state，下游可取用
        request.state.language = lang
        return await call_next(request)
```

或直接在需要的接口用 `Header` 依赖注入：

```python
from fastapi import Header

async def rag_query_handler(
    x_language: str = Header(default="zh-CN", alias="X-Language"),
    ...
):
    ...
```

### 4.2 Query 翻译工具

**新增文件**：`app/modules/rag/translation.py`

```python
"""
Query 翻译工具：将非中文 query 翻译为中文，供 RAG 检索使用。
使用小模型（如 claude-haiku / gpt-4o-mini）以降低延迟和成本。
"""

from app.config import settings

TRANSLATE_PROMPT = (
    "Translate the following text to Simplified Chinese. "
    "Output ONLY the translation, no explanation.\n\nText: {text}"
)

async def translate_to_zh(text: str, source_lang: str) -> str:
    """
    将 text 翻译为中文。
    如果 source_lang 已是 zh-CN，直接返回原文。
    """
    if source_lang == "zh-CN":
        return text

    # 使用已有 LLM 客户端（复用 app/modules/llm/ 的基础设施）
    from app.modules.llm.services.completion_service import quick_complete
    prompt = TRANSLATE_PROMPT.format(text=text)
    translated = await quick_complete(prompt, model=settings.TRANSLATION_MODEL)
    return translated.strip()
```

> `quick_complete` 为 LLM 模块中封装的单次同步调用函数（需按实际情况实现或复用现有接口）。
> `settings.TRANSLATION_MODEL` 在 `.env` 中配置，默认指向速度快、成本低的模型。

### 4.3 RAG Query 入口改造

**文件**：`app/modules/ppt/handlers/rag_query/orchestrator.py`

在检索步骤前插入翻译：

```python
async def run(self, query: str, language: str = "zh-CN", ...):
    # 1. 翻译 query（非中文时）
    search_query = await translate_to_zh(query, source_lang=language)

    # 2. RAG 检索（使用翻译后的 query）
    chunks = await self.search(search_query, ...)

    # 3. LLM 生成（原始 query + 中文 chunks + 语言指令）
    await self.generate(
        user_query=query,          # 原始语言，保持用户意图
        chunks=chunks,
        language=language,
        ...
    )
```

### 4.4 LLM System Prompt 语言匹配

**文件**：`app/modules/ppt/handlers/rag_query/orchestrator.py`（或 prompt 构建处）

```python
LANGUAGE_INSTRUCTION = {
    "zh-CN": "",                              # 中文无需额外指令
    "en-US": "Please respond in English.",
    "ja-JP": "Please respond in Japanese.",
}

def build_system_prompt(base_prompt: str, language: str) -> str:
    instruction = LANGUAGE_INSTRUCTION.get(language, "")
    if instruction:
        return f"{base_prompt}\n\n{instruction}"
    return base_prompt
```

## 七、注意事项

### 7.1 翻译延迟

Query 翻译是额外的 LLM 调用，会增加首 token 延迟（通常 200~500ms）。
缓解措施：
- 使用轻量模型（Haiku、GPT-4o-mini）
- 翻译与其他前置操作并行（如 conversation 初始化）

### 7.2 专有名词翻译偏差

用户 query 中若包含专有名词（产品名、人名等），LLM 翻译可能引入偏差影响检索。
当前方案接受此风险，后续可通过"术语表"注入翻译 prompt 改善。

### 7.3 新增语言扩展方式

扩展新语言只需：
1. `LOCALES` 常量新增语言包
2. `LANGUAGE_INSTRUCTION` 新增语言指令
3. 前端语言切换逻辑改为下拉选择（替代当前双语 toggle）

### 7.4 不影响现有功能

- RAG 知识库 embedding、chunk 结构、Meilisearch 索引**不需要任何变更**
- 中文用户路径完全不走翻译逻辑，无性能影响
